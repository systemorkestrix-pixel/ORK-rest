# Telegram Delivery Bot Operations Runbook

تاريخ التحديث: `2026-03-29`

## الهدف
- تشغيل بوت التوصيل بثبات أثناء التطوير المحلي.
- تسهيل تشخيص الأعطال السريعة دون الرجوع إلى الكود كل مرة.
- توضيح أدوات الدعم الموجودة الآن داخل اللوحة.

## الحالة الحالية
- البوت يعمل من نفس `backend` وليس كخدمة مستقلة.
- مسار الـ webhook:
  - `/api/bot/telegram/<webhook_secret>`
- الربط الحالي المحلي يعتمد على `cloudflared quick tunnel`.
- السائق يتعامل يوميًا عبر Telegram.
- لوحة الويب الخاصة بالتوصيل تبقى احتياطية فقط.

## المتطلبات الأساسية
- تشغيل `backend` على `8124`
- تشغيل الواجهة عبر `npm run dev` أو بيئة التشغيل المعتادة
- وجود:
  - `bot token`
  - `bot username`
  - `webhook_secret`
  داخل إعدادات النظام

## التشغيل المحلي السريع
1. شغّل النظام:
   - `npm run dev`
2. افتح نفقًا عامًا إلى الـ backend:
   - `cloudflared tunnel --url http://127.0.0.1:8124`
3. انسخ رابط `trycloudflare.com` الجديد.
4. حدّث Telegram webhook على هذا الرابط.

الصيغة النهائية للـ webhook:

```text
https://<trycloudflare-domain>/api/bot/telegram/<webhook_secret>
```

## أوامر تحديث webhook
في PowerShell:

```powershell
$botToken="PUT_REAL_BOT_TOKEN_HERE"
$webhookUrl="https://<trycloudflare-domain>/api/bot/telegram/<webhook_secret>"

Invoke-RestMethod -Uri "https://api.telegram.org/bot$botToken/deleteWebhook?drop_pending_updates=true"
Invoke-RestMethod -Uri "https://api.telegram.org/bot$botToken/setWebhook?url=$([uri]::EscapeDataString($webhookUrl))"
Invoke-RestMethod -Uri "https://api.telegram.org/bot$botToken/getWebhookInfo"
```

## ما الذي يجب أن أراه بعد الربط الصحيح
- `url` يطابق رابط `trycloudflare` الحالي
- `pending_update_count = 0`
- لا يوجد `last_error_message`

## أعطال شائعة وطريقة التعامل

### 1. البوت لا يرد على `/start`
راجع بالترتيب:
- هل `cloudflared` ما زال مفتوحًا؟
- هل `getWebhookInfo` يشير إلى نفس الرابط الحالي؟
- هل تغيّر رابط `trycloudflare` بعد إعادة تشغيل `cloudflared`؟
- هل السائق استعمل رابط ربط قديم أو منتهي الصلاحية؟

الإجراء:
- أعد ربط webhook
- أنشئ رابط ربط جديد للسائق من اللوحة

### 2. الأزرار لا تستجيب أو يبدو البوت جامدًا
راجع:
- `pending_update_count`
- `last_error_message`
- هل Telegram يرسل إلى رابط tunnel قديم؟

الإجراء:
- أعد تشغيل `cloudflared`
- أعد تسجيل webhook
- استخدم عرضًا أو مهمة جديدة بدل الرسائل القديمة

### 3. `404` على مسارات دعم Telegram من اللوحة
مثال:
- `/api/manager/drivers/{id}/telegram-link/test-message`
- `/api/manager/drivers/{id}/telegram-link/resend-latest`

المعنى:
- غالبًا الـ backend الحالي لم يحمل آخر نسخة من الكود

الإجراء:
- أوقف `npm run dev`
- تأكد من عدم وجود عملية قديمة على `8124`
- شغّل `npm run dev` من جديد

### 4. السائق مربوط لكن لا تصله الرسائل
راجع:
- حالة البوت من لوحة الإعدادات
- حالة السائق من شاشة إدارة السائق
- هل `telegram_chat_id` و`telegram_enabled` موجودان؟

الإجراء:
- استخدم `رسالة اختبار`
- إذا فشلت، افصل الربط ثم أنشئ رابطًا جديدًا

## أدوات الدعم الموجودة الآن في اللوحة
من شاشة إدارة السائق:
- `إنشاء رابط ربط`
- `فك الربط`
- `رسالة اختبار`
- `إعادة إرسال آخر مهمة/عرض`

كما تعرض الشاشة:
- حالة الربط
- اسم مستخدم Telegram
- الجهة الحالية
- هل لدى السائق مهمة جارية
- هل لديه عرض مفتوح
- تلميح استرداد سريع

## متى نستخدم كل أداة

### إنشاء رابط ربط
- عند أول ربط للسائق
- أو عند انتهاء صلاحية الرابط القديم

### فك الربط
- عند تغيير هاتف السائق
- عند تبديل الحساب
- عند الاشتباه أن الربط ذهب لسائق آخر

### رسالة اختبار
- للتحقق السريع أن البوت ما زال يصل إلى السائق
- بعد تغيير webhook
- بعد تغيير token أو username

### إعادة إرسال آخر مهمة/عرض
- إذا قال السائق إن الرسالة اختفت
- إذا تغيّر tunnel أثناء وجود عرض مفتوح
- إذا احتجنا إعادة إظهار المهمة الحالية بسرعة

## قواعد التشغيل الحالية
- السائق لا يجب أن يملك أكثر من مهمة نشطة واحدة في نفس الوقت.
- إذا التقط السائق طلبًا، تُسحب العروض الأخرى المفتوحة له.
- إذا فشل التوصيل:
  - تُزال المهمة من المهام الجارية
  - وتنتقل المعالجة التالية إلى الإدارة

## ملاحظات إنتاجية لاحقة
- رابط `trycloudflare` للتطوير فقط.
- في الإنتاج يجب استخدام رابط عام ثابت.
- خطوة ربط `base URL` بمتغيرات البيئة مؤجلة إلى مرحلة تنظيف المشروع وتهيئة الإنتاج.

## قرار الإغلاق الحالي
- هذا الجانب مغلق تشغيليًا على التطوير المحلي.
- أي مشكلة لاحقة في البوت يجب أن تبدأ من هذا الترتيب:
  1. فحص حالة البوت من الإعدادات
  2. فحص webhook
  3. تجربة `رسالة اختبار`
  4. تجربة `إعادة إرسال آخر مهمة/عرض`
